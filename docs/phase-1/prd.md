# DropRoom 一期产品需求文档

## 1. 文档信息

- 阶段：一期（MVP）
- 状态：一期实现完成，待实机兼容性和生产环境验收
- 产品定位：以临时房间为核心的多人 Web 文件传输助手
- 核心场景：用户通过公网服务器，在多台设备或多个人之间快速传递文字、图片和文件

## 2. 产品原则

1. DropRoom 是传输工具，不是聊天产品。
2. 用户无需注册账号，进入房间后即可使用。
3. 房间是内容、成员、权限和生命周期的唯一边界。
4. 服务端仅在房间存续期间保存内容，房间销毁后不保留数据。
5. 优先使用浏览器原生能力，减少额外操作和兼容性问题。

## 3. 一期目标

- 用户可以一键创建临时房间，并复制邀请链接给其他设备或成员。
- 房间成员可以传递文字、图片和普通文件。
- 新加入的成员可以看到当前房间生命周期内已有的内容。
- 文件上传、容量、失败状态和房间有效期对用户清晰可见。
- 房间到期或无人使用后，服务端自动清理全部内容。

## 4. 非目标

一期不提供以下能力：

- 注册、登录和长期用户账号
- 私聊、回复、引用、表情、已读状态和接收确认
- 好友、群组、联系人和社交关系
- 永久聊天记录或跨房间内容检索
- 网页内接管下载目录或自定义下载器
- 端到端加密、病毒扫描和高级反滥用能力
- 浏览器指纹识别
- 多房间列表、房间收藏和房间间快速切换

## 5. 用户与身份

### 5.1 临时身份

- 用户无需注册或登录。
- 用户可以设置并随时修改昵称。
- 昵称允许重复，不作为权限或身份判断依据。
- 每名成员在当前房间内拥有一个数字 ID，用于区分同名成员。
- 浏览器在本地保存最近使用的昵称，后续进入房间时自动带出。
- 一期不使用浏览器指纹。

### 5.2 房主身份

- 创建房间的用户是初始房主。
- 房主权限通过独立随机成员凭证识别，不依赖昵称。
- 房主可以修改房间名称、删除房间内任意文件以及主动解散房间。
- 普通成员只能删除自己发送的文件。

## 6. 房间

### 6.0 单房间使用

- 首页只提供创建房间和输入房间码加入两个入口。
- 进入房间后，界面只展示当前房间的内容、成员、容量和生命周期。
- 一期不提供房间列表和房间间快速切换；用户需要进入其他房间时，返回首页重新创建或加入。
- 离开当前页面不等于解散房间，房间仍遵循5分钟空房清理和24小时强制销毁规则。
- 房间内点击品牌 Logo 返回首页时，必须复用退出房间确认和主动退出流程。

### 6.1 创建房间

- 用户点击创建后立即生成房间，并使用自动生成的默认名称。
- 创建成功后生成一个8位全随机数字房间码。
- 房间码允许前导零，界面可以分组展示，例如 `0123 4567`。
- 房间码在房间存续期间保持不变，不允许重新生成。
- 当前存活的房间码必须唯一。
- 房主可以修改自动生成的房间名称，但修改名称不影响房间码。

### 6.2 加入房间

- 用户输入完整房间码即可加入。
- 新浏览器打开邀请链接后展示加入确认页，用户可以先修改昵称，再加入对应房间。
- 输入时忽略展示用空格。
- 同一 IP 每分钟最多进行10次失败的加入尝试。
- 超过限制后，界面应显示暂时无法继续尝试以及剩余等待时间。
- 成功加入不作为失败次数累计。

### 6.3 成员数量

- 每个房间最多保留9名成员，房主计入人数。
- 在线成员和处于5分钟断线重连宽限期内的成员都会占用名额；宽限期结束并清理成员后释放名额。
- 达到上限后，新成员无法加入，并收到“房间已满”的明确提示。
- 房间需要展示当前在线成员数量。

### 6.4 房主转移

- 房主主动退出时，房主权限立即转移给当前仍在线且加入顺序最早的成员。
- 房主刷新页面或意外断线时，保留房主身份5分钟。
- 房主在5分钟内重连则恢复原有身份。
- 房主超过5分钟未重连且房间内仍有其他成员时，房主权限转移给当前加入顺序最早的成员。

### 6.5 房间销毁

- 房间从创建时开始计算，最长存续24小时，不能续期。
- 即使仍有成员在线，达到24小时后也必须销毁。
- 最后一名成员离线后进入5分钟宽限期；宽限期内有人重连则取消销毁。
- 连续5分钟无人在线时，房间自动销毁。
- 空房清理用于控制个人服务器的磁盘和带宽成本，不能作为长期暂存能力使用。
- 房主主动解散房间时立即销毁，不进入宽限期。
- 房间临近24小时到期时需要向在线成员展示明显提醒和倒计时。
- 房间销毁后，文字历史、文件元信息、暂存文件和房间状态全部删除。

## 7. 内容传输

### 7.1 文字

- 成员可以发送纯文字内容。
- 文字作为房间历史保留至房间销毁。
- 新加入或重新加载页面的成员可以看到当前房间内已有文字。
- 文字不计入文件容量。
- 一期不提供编辑、回复、已读和接收确认。
- 文字提交期间发送按钮显示加载状态并阻止重复提交；失败后保留输入内容。

### 7.2 文件

- 成员可以选择、粘贴或拖放并同时发送多个文件。
- 房间内只有一名成员在线时，也可以上传文件。
- 文件开始上传后，房间内其他成员可以立即看到文件卡片和上传状态。
- 文件上传完成后，其他成员可以随时点击下载，发送设备无需保持在线。
- 文件由服务器暂存到房间销毁为止，不作为永久数据保存。
- 单个文件失败不影响同一批次内其他文件。
- 大文件使用固定大小的分片上传。
- 网络中断后保留已经完成的分片，并自动重试当前分片。
- 当前页面仍持有原始文件时，可以直接继续上传。
- 页面刷新后，用户重新选择同一个文件，可以从已完成位置继续上传。
- 用户取消上传后，服务端删除临时分片并释放容量。

### 7.3 图片

- 图片作为特殊文件处理，遵循相同的上传、容量和生命周期规则。
- 图片上传完成后显示小尺寸预览。
- 用户可以点击查看原图或使用浏览器下载。
- 一期不提供图片编辑、压缩和相册功能。

### 7.4 文件状态

文件卡片至少包含以下状态：

- 上传中，并显示当前进度
- 上传完成，可以下载
- 上传失败
- 已取消
- 已删除或不可用

不展示每位成员是否已经下载或接收。

## 8. 容量规则

- 单次选择、粘贴或拖放的一批文件总大小不得超过300 MB。
- 不另设单文件大小配置；单个文件作为一批上传时同样不得超过300 MB。
- 一个房间内暂存文件总量不得超过1 GB。
- 文字不计入文件容量。
- 上传中的文件需要预占容量，避免并发上传突破1 GB上限。
- 上传失败、取消或文件被删除后，释放对应容量。
- 页面持续展示房间文件的已使用容量和剩余容量。
- 前端容量数值和提示必须读取服务端房间快照中的当前配置，不固定写死默认值。
- 容量不足时，需要在上传开始前给出明确提示。

## 9. 上传与下载体验

### 9.1 上传

- 每个文件独立显示上传进度、成功或失败状态。
- 小文件使用单一总体进度条；多分片文件同时展示文件总体进度和当前分片进度。
- 多文件上传时，一个文件失败不阻塞其他文件。
- 等待并发名额的文件显示“排队中”，中断后显示“上传已暂停”。
- 空文件在选择后直接给出不支持提示，不创建上传任务。
- 在文字输入框粘贴图片或文件时直接进入文件上传流程；粘贴纯文字仍写入输入框。
- 上传中断后允许用户继续上传或重新选择同一个文件恢复任务。
- 发送设备在上传完成前离线时，其他成员只能看到当前进度，不能下载未完成文件。

### 9.2 下载

- 文件下载遵循标准 HTTP 和浏览器原生下载行为。
- 文件保存目录、下载确认和下载进度由浏览器负责。
- 网页不尝试直接写入用户指定目录，也不重复实现下载进度界面。

## 10. 历史记录与本地存储

- 文字历史和文件列表以当前房间服务端状态为准。
- 新成员加入或页面刷新后，从服务器获取房间当前内容。
- IndexedDB 不承担房间历史记录的主要责任。
- 浏览器本地仅保存昵称、房间恢复所需的轻量信息和必要偏好。
- 成员凭证使用每个房间独立的 HttpOnly Cookie，不写入 URL 或本地存储。
- 房间销毁后，用户不能通过重新加入恢复历史或文件。

## 11. 关键异常场景

- 房间码不存在、已过期或已经销毁。
- 房间人数达到9人。
- 同一 IP 的失败加入尝试超过限制。
- 单批文件超过300 MB。
- 房间容量达到或即将超过1 GB。
- 分片上传过程中发送设备断线，或重新选择的文件与原任务不匹配。
- 房间在上传或下载期间达到24小时有效期。
- 房主主动退出、意外断线或解散房间。
- 多个成员同时上传导致容量竞争。

所有异常都应提供明确结果，避免让用户误以为文件已经成功保存或传输。

## 12. 一期验收标准

一期满足以下条件时可以视为达到产品目标：

1. 用户无需账号即可一键创建房间，并通过房间码或邀请链接加入。
2. 最多9名成员可以查看实时内容，同名成员可以通过数字 ID 区分。
3. 新加入成员可以看到房间存续期间已有的文字和已上传文件。
4. 文字、普通文件和图片均可按定义传输和展示。
5. 单批300 MB、单房间1 GB和单实例30 GB容量限制能够正确执行。
6. 上传进度、失败状态、已用容量、剩余容量和房间到期提醒清晰可见。
7. 上传中断后可以从已完成分片继续，上传完成后发送设备离线不影响下载。
8. 房主转移、主动解散、5分钟无人销毁和24小时强制销毁行为符合定义。
9. 房间销毁后，服务端不再保留或提供该房间的文字和文件。
10. 成员凭证不出现在 SSE、预览和下载 URL 中，也不写入本地存储。

## 13. 一期实现约定

- API 默认使用不常用端口 `43117`。
- 单条文字最大20,000字符。
- 单批最多50个文件，文件大小合计不超过300,000,000字节；不另设单文件大小配置。
- 单房间文件总量不超过1,000,000,000字节。
- 上述限制是默认值，生产部署可以通过环境变量覆盖；服务端必须将当前限制下发给前端并作为最终校验方。
- 每名成员最多同时进行3个实际文件上传请求，更多文件在前端排队。
- 每个上传分片默认不超过2,000,000字节，并使用 SHA-256校验。
- 服务端根据已接收字节数校验续传位置。
- 创建房间时从各20个内置形容词和名词中随机组合默认名称，共400种组合；房主创建后可以修改。
- 同一 IP 每10分钟最多创建10个房间。
- 300 MB、1 GB和30 GB采用十进制字节口径。
- 房间到期时间由服务端返回 `expiresAt`，前端据此显示倒计时；剩余30分钟显示顶部提醒，剩余5分钟显示不可关闭的强提醒。
- 一期房间状态仍采用单实例内存状态与本地临时文件；服务进程重启视为所有房间销毁。
- 创建房间和失败加入限流使用 Redis 保存滑动窗口，避免 API 重启后立即清空限流记录。
- 本地 Redis 默认使用不常用端口 `46379`，生产环境必须配置真实的 `DROPROOM_REDIS_URL`。
- 服务启动时清理专用临时目录，房间销毁时清理对应文件目录。
- 单实例默认设置30,000,000,000字节的全局临时文件安全上限。
- 前后端写操作使用 HTTP，房间实时事件使用 SSE，文件下载使用浏览器原生 HTTP。
- Web 通过每个房间独立的 HttpOnly Cookie 访问 SSE、预览和下载地址。
- 接口日志使用 Pino，所有请求包含 `X-Request-Id`、状态码和耗时，成员凭证自动脱敏。
- 服务提供从 Zod请求与响应模型生成的 OpenAPI 3.1规范和 Swagger UI。
- 默认文档地址为 `/openapi.json` 和 `/docs`，生产环境可以通过环境变量关闭。
- 环境变量使用 `dotenv` 加载并通过 Zod统一校验；本地开发无需填写配置，生产环境强制填写 `NODE_ENV=production`、真实的 `DROPROOM_CORS_ORIGINS` 和 `DROPROOM_REDIS_URL`。
- API 使用多阶段 Dockerfile 构建非 root 生产镜像，通过 `tsx` 直接启动 TypeScript
  源码，镜像发布到 `boses/droproom-api`。
- Web 使用 Vite 构建静态资源并由 Nginx 提供服务，生产环境通过同源 `/api`
  反向代理访问后端，镜像发布到 `boses/droproom-web`。
- Vite 使用 Rolldown 按 React、Ant Design、TanStack 和其他依赖拆分生产资源；
  Nginx 为哈希资源配置长期缓存和 gzip，为 SPA 配置路由回退，并为 SSE 与大文件
  上传配置代理缓冲、请求体和超时限制。
- GitHub Actions 在推送两个镜像前执行前后端 lint、类型检查、测试和构建。
- 一期最低支持 Chrome 111、Edge 111、Firefox 114、Safari 16.4 和 iOS/iPadOS Safari 16.4，构建目标固定在 Vite 8 Baseline Widely Available。
